# LifeOS AI 识别规则

> 目的：把 AI 归类边界、二次确认标准和回归样本维护方式固定下来。这里是产品契约；Worker prompt、iOS 端落库策略和测试集都应向这份文档对齐。

## 1. 基本原则

- **先接住，再整理**：AI 识别的第一目标是让用户输入不丢失，不追求一次性完美整理。
- **明确才自动执行**：只有明确时间段、明确待办语义或明确记录类型时，才自动进入对应模块。
- **模糊内容保守处理**：带评价、偏好、感受、复盘的输入，默认进入随手记观察，不默认变成待办。
- **混合输入允许拆分**：同一句里既有时间事实，也有复盘想法时，可以拆成多条记录。
- **低压力**：任何不确定都不使用警告式文案，不制造 streak、完成率或失败感。

## 2. 模块边界

### 时间记录

进入时间记录需要满足至少一条：

- 原文有明确起止时间，例如 `10 点到 11 点`、`09:30-11:00`。
- AI record 已解析出有效 `startTime` 和 `endTime`，即使原文只写了一个钟点和时长描述。
- Apple 健康或用户手动操作提供了明确时间段。

不进入时间记录：

- 只有单个时间点且无时长，例如 `11 点面试完`。
- AI 返回 `startTime == endTime` 的零长度时间段，例如 `11:00-11:00`。
- 未来安排类表达更像待办/日程，例如 `今天下午我有一场面试，2 到 3 点`。

### 待办

进入待办需要明确执行意图，例如：

- `待办`、`todo`、`提醒我`、`记得`、`加入待办`。
- 有明确需要执行的动作和时间，例如 `明天提交报销`。

不默认进入待办：

- 愿望、偏好、评价、复盘判断，例如 `希望找更年轻有活力的团队`。
- 感受类输入，例如 `感觉很焦虑`。
- 明确说要放到随手记/想法/感受的内容。

### 随手记

随手记用于接住观察、想法、感受、感恩、做梦，以及暂时无法确定的内容。

- `想法`：判断、偏好、复盘、计划雏形、尚未形成行动的念头。
- `感受`：情绪、身体状态、关系中的主观体验。默认适合观察和沉淀，不默认转待办。
- `感恩`：明确感谢、感恩内容。
- `做梦`：梦境、睡醒后的梦内容。

## 3. 二次确认标准

这轮只定义标准，不实现新的三选确认 UI。

需要二次确认的典型情况：

- AI 认为像待办，但原文没有强待办信号。当前策略是先保守放到随手记，再提供“改成待办”。
- 同一句同时含时间事实和复盘感受，且 AI 无法稳定拆分。
- 原文有未来时间和动作，但既可能是日程，也可能只是回忆/复盘。

不需要二次确认：

- 明确时间段已经解析成功。
- 明确待办信号已经出现。
- 明确感受/感恩/做梦类型。

## 4. DBT Coach 边界

DBT Coach 不是普通记录归类，也不是二次确认 UI；它是 AI 对话里的独立练习状态机。

- 普通 chat 模式可以在情绪调节、反复纠结、人际困扰、行为复盘等场景里提议 DBT 练习。
- 用户未同意前，不直接开始 DBT 步骤，也不生成 DBT 练习总结。
- 用户同意后，App 端应确定性启动 `AgentDBTSessionState`，并切到 `agentMode = "dbtCoach"`。
- DBT Coach 每轮最多推进一个步骤，未完成时不生成 action card。
- 具体 handoff、步骤推进和回归样本见 [`agent-chat-and-dbt-coach.md`](agent-chat-and-dbt-coach.md)。

## 5. 标准测试集维护

- 标准样本放在 `Tests/Fixtures/ai-routing-cases.json`。
- 每个 case 至少包含 `id`、`input`、`aiRecords`、`expectedRecords`、`notes`、`tags`。
- 修复 AI 归类 bug 时，先把失败输入加入样本库，再改策略。
- 样本库只放脱敏文本；不要放真实凭证、联系方式、地址或不可公开的隐私内容。
- `expectedRecords` 只描述 iOS 端最终应落到哪里，不绑定远端模型的中间推理。
- 每次新增样本时优先复用已有 `tags`，只有出现稳定的新边界时再扩展标签。
- 如果一个真实输入同时包含时间事实、情绪感受和偏好判断，样本应拆清楚期望：时间记录不能丢，感受/偏好默认进入随手记，不直接转待办。

维护流程：

1. 从 AI 调试记录、用户反馈或手动复现中提炼失败输入。
2. 先写入 JSON 样本，并在 `notes` 说明为什么这个 case 重要。
3. 再修改 `AIRoutingPolicy` 或相关解析逻辑。
4. 跑完整单测，确认新增样本通过且旧样本没有回归。
5. 如果这次改动改变了产品边界，同步更新本文档前面的规则章节。

## 6. 当前不做

- 不新增真正数据库。
- 不改 Worker prompt。
- 不实现多 agent 框架。
- 不新增陪伴型连续对话入口。
- 不新增新的三选确认 UI。
